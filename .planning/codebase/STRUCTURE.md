# Codebase Structure

**Analysis Date:** 2026-04-09

## Directory Layout

```
homebridge-neakasa/
├── src/                       # All TypeScript source code
│   ├── index.ts              # Plugin entry point; registers platform
│   ├── platform.ts           # NeakasaPlatform class; orchestrates discovery and polling
│   ├── accessory.ts          # NeakasaAccessory class; manages HomeKit services per device
│   ├── api.ts                # NeakasaAPI class; handles Neakasa cloud authentication and requests
│   ├── client.ts             # IoTClient class; HMAC request signing and nonce management
│   ├── encryption.ts         # AES-128-CBC encryption utilities for tokens
│   ├── types.ts              # All TypeScript interfaces and enums
│   └── settings.ts           # Constants (PLATFORM_NAME, etc.)
├── test/                      # Jest test suite
│   ├── accessory-helpers.test.ts     # Tests for accessory service creation and updates
│   ├── encryption.test.ts            # Tests for encryption utilities
│   └── platform-config.test.ts       # Tests for configuration validation and merging
├── dist/                      # Compiled JavaScript output (generated by build)
├── jest.config.js            # Jest test runner configuration
├── tsconfig.json             # TypeScript compiler configuration
├── .eslintrc                 # ESLint configuration
├── package.json              # NPM dependencies and scripts
├── package-lock.json         # Lock file for dependencies
├── .npmignore                # Files to exclude from npm package
├── config.schema.json        # Homebridge config schema for UI
├── CLAUDE.md                 # Developer guide with architecture overview
├── README.md                 # Project documentation
├── QUICKSTART.md             # Setup instructions
├── CHANGELOG.md              # Version history
├── SWITCH_SENSOR_DETAILS.md  # Documentation of HomeKit service/sensor mappings
├── LICENSE                   # MIT or similar
└── branding/                 # Icons and branding assets
    └── icon.png              # Plugin icon for Homebridge UI
```

## Directory Purposes

**src/ — TypeScript Source**
- Purpose: All plugin implementation code
- Contains: Entry point, platform lifecycle, accessory management, API communication, encryption, type definitions
- Key files: `platform.ts`, `accessory.ts`, `api.ts`

**test/ — Jest Test Suite**
- Purpose: Unit and integration tests for core functionality
- Contains: Test files matching `*.test.ts` pattern
- Key files: `platform-config.test.ts` (config validation), `accessory-helpers.test.ts` (service creation), `encryption.test.ts` (crypto)

**dist/ — Compiled Output**
- Purpose: Generated JavaScript from TypeScript compilation
- Contains: `*.js` and `*.js.map` files (created by `npm run build`)
- Generated: Yes (not committed)
- Committed: No

**branding/ — Assets**
- Purpose: Icon and branding materials for Homebridge UI
- Contains: PNG icons
- Key files: `icon.png`

## Key File Locations

**Entry Points:**
- `src/index.ts`: Plugin registration with Homebridge; imports and registers `NeakasaPlatform`
- `src/platform.ts`: `NeakasaPlatform` class lifecycle (constructor, didFinishLaunching, configureAccessory)

**Configuration:**
- `config.schema.json`: Homebridge UI configuration schema; defines user-facing options
- `tsconfig.json`: TypeScript compilation target (ES2020, module: commonjs, strict mode enabled)
- `jest.config.js`: Test runner config (ts-jest preset, Node environment, test file patterns)
- `.eslintrc`: Linting rules (single quotes, 2-space indent, 140-char line max, allow no-non-null-assertion and no-explicit-any)

**Core Logic:**
- `src/platform.ts`: Device discovery, polling orchestration, config merge/validation
- `src/accessory.ts`: HomeKit service composition, characteristic handlers
- `src/api.ts`: Authentication flow, device property/action management
- `src/client.ts`: HMAC signing, request/response handling

**Utilities:**
- `src/encryption.ts`: AES-128-CBC encryption for token handling
- `src/types.ts`: All TypeScript type definitions
- `src/settings.ts`: Constants and configuration metadata

**Testing:**
- `test/platform-config.test.ts`: Tests for `sanitizeConfig()` and config merge precedence
- `test/accessory-helpers.test.ts`: Tests for service creation logic and feature flags
- `test/encryption.test.ts`: Tests for AES encryption/decryption

## Naming Conventions

**Files:**
- Kebab-case for test files: `accessory-helpers.test.ts`, `platform-config.test.ts`
- Camelcase for implementation: `platform.ts`, `accessory.ts`, `api.ts`
- All TypeScript: `.ts` extension

**Directories:**
- Lowercase: `src/`, `test/`, `dist/`, `branding/`

**Class Names (TypeScript):**
- PascalCase: `NeakasaPlatform`, `NeakasaAccessory`, `NeakasaAPI`, `IoTClient`

**Function/Method Names:**
- camelCase: `sanitizeConfig()`, `updateCharacteristics()`, `invokeService()`, `getDeviceData()`

**Constants:**
- UPPER_SNAKE_CASE: `PLATFORM_NAME`

**Types/Interfaces:**
- PascalCase: `DeviceData`, `PlatformConfig`, `ServiceConfig`, `AuthResponse`

**Enums:**
- PascalCase for enum name: `BucketStatus`, `RoomOfBin`
- UPPER_SNAKE_CASE for values: `BUCKET_STATUS_CLEAN`, `ROOM_OF_BIN_LIVING_ROOM`

## Where to Add New Code

**New Feature (e.g., additional HomeKit service type):**
- Primary implementation: `src/accessory.ts` (add service creation logic and characteristic handlers)
- Type definitions: `src/types.ts` (add device property or config option)
- Tests: `test/accessory-helpers.test.ts` (test service creation for new feature)
- Config schema: `config.schema.json` (add feature flag if user-configurable)

**New Device Action (e.g., new command to send to device):**
- API implementation: `src/api.ts` (add method to `NeakasaAPI`)
- Accessory handler: `src/accessory.ts` (add characteristic handler that calls API method)
- Type definitions: `src/types.ts` (add request/response types if needed)
- Tests: `test/accessory-helpers.test.ts` (mock API and test handler)

**New Utilities/Helpers:**
- Shared encryption logic: `src/encryption.ts`
- Shared types: `src/types.ts`
- Request signing helpers: `src/client.ts`

**New Test:**
- Location: `test/` directory
- Naming: `[feature].test.ts` (e.g., `weight-sensors.test.ts`)
- Pattern: Use Jest, match existing test structure (describe blocks, mocking patterns)

**Configuration Changes:**
- User-facing options: Add schema to `config.schema.json`
- Type safety: Add to appropriate interface in `src/types.ts`
- Validation logic: Update `sanitizeConfig()` in `src/platform.ts`
- Documentation: Update `SWITCH_SENSOR_DETAILS.md` if user-impacting

## Special Directories

**dist/ (Compiled Output):**
- Purpose: Contains `*.js` and `*.js.map` files compiled from TypeScript
- Generated: Yes, via `npm run build` (runs `tsc`)
- Committed: No (in `.npmignore` and `.gitignore`)
- Consumed by: Homebridge when loading plugin via package.json "main" field

**node_modules/ (Dependencies):**
- Purpose: NPM package dependencies
- Generated: Yes, via `npm install`
- Committed: No (large; use package-lock.json for reproducibility)

## Build & Compilation

**Build Outputs:**
- Input: `src/**/*.ts` and `src/**/*.ts` files
- Output: `dist/` directory with compiled `.js` files
- Command: `npm run build` (or `tsc` directly)
- Mode: Production build; includes sourcemaps for debugging

**Source Maps:**
- Generated: Yes (`dist/**/*.js.map`)
- Used for: Stack trace debugging when plugin errors occur in Homebridge

**Package Entry Point:**
- Defined in: `package.json` "main" field (typically `"dist/index.js"`)
- Loaded by: Homebridge when plugin starts

## Configuration Files

**config.schema.json:**
- Purpose: Defines Homebridge UI form for plugin configuration
- Structure: JSON schema describing nested config object
- Contains: Platform name, credential fields, feature flags, per-device overrides

**tsconfig.json:**
- Purpose: TypeScript compilation settings
- Key settings: ES2020 target, commonjs modules, strict null checks, no implicit any
- Declaration: Generates `.d.ts` files for type definitions

**jest.config.js:**
- Purpose: Jest test runner configuration
- Settings: ts-jest preset, test roots, test match patterns
- Coverage: Jest can generate coverage reports (typically in `coverage/` directory)

**package.json:**
- Purpose: Define project metadata, dependencies, and npm scripts
- Scripts: `build`, `lint`, `test`, `watch`
- CI: Runs lint → build → test on Node 18 and 20

## Import Path Organization

**Imports (Expected Pattern):**
1. External packages (homebridge, node built-ins)
2. Local modules from `src/`
3. Types imported with `import type` for type-only imports

**Example (from src/platform.ts):**
```typescript
import { API, DynamicPlatformPlugin, Logger, PlatformConfig } from 'homebridge';
import { PLATFORM_NAME } from './settings';
import { NeakasaAccessory } from './accessory';
import { NeakasaAPI } from './api';
import type { DeviceData, PlatformConfig as CustomConfig } from './types';
```
